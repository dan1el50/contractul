# Development setup

How to run the stack locally. Everything runs in Docker — you do not install Python,
Node, PostgreSQL, or LibreOffice on your machine.

> **Target state.** The Docker setup described here does not exist yet; it is the
> specification for the branch that builds it. Until then, the design prototype is opened
> by loading the `*.dc.html` files directly in a browser — no tooling required.

## Prerequisites

| Tool | Version | Notes |
| --- | --- | --- |
| Docker Desktop | current | Includes Compose v2. On Windows, back it with WSL 2. |
| Git | current | |

That is the whole list, and it is the point of containerising. If a fifth thing appears in
this table, something has gone wrong.

An editor with Python and TypeScript support helps, but nothing about the stack depends on
your choice of one.

## First run

```bash
git clone <repository-url>
cd contractul

cp .env.example .env        # Windows PowerShell: Copy-Item .env.example .env

docker compose up --build
```

The first build takes several minutes — it installs LibreOffice, which is large. Later
builds are cached and fast.

When it settles:

| Service | URL |
| --- | --- |
| Frontend | http://localhost:5173 |
| API | http://localhost:8000 |
| API documentation (OpenAPI) | http://localhost:8000/docs |
| PostgreSQL | `localhost:5432` |

Apply migrations and load sample data:

```bash
docker compose exec backend alembic upgrade head
docker compose exec backend python -m app.scripts.seed
```

The seed data mirrors the design prototype — the same catalog, the same demo user — so
what you see locally matches the screens the product was designed against.

## Everyday commands

```bash
docker compose up                  # start
docker compose up -d               # start detached
docker compose down                # stop
docker compose logs -f backend     # follow one service's logs
docker compose ps                  # what is running
```

Both applications hot-reload. Editing backend or frontend source is reflected without a
restart.

**Rebuild when dependencies change.** Adding a package to `pyproject.toml` or
`package.json` requires `docker compose up --build`. Hot reload watches your source, not
your dependency manifest — a mysteriously missing import is almost always this.

### Backend

```bash
docker compose exec backend bash                       # a shell in the container

docker compose exec backend pytest                     # all tests
docker compose exec backend pytest tests/unit          # just the fast ones

docker compose exec backend alembic upgrade head       # apply migrations
docker compose exec backend alembic revision --autogenerate -m "add wallet"
```

**Always read an autogenerated migration before committing it.** Alembic infers intent
from a schema diff and it infers imperfectly — renames in particular tend to come out as a
drop plus an add, which is data loss wearing a convincing costume.

### Frontend

```bash
docker compose exec frontend sh

docker compose exec frontend npm run test
docker compose exec frontend npm run lint
docker compose exec frontend npm run build      # production build, catches type errors
```

### Database

```bash
docker compose exec database psql -U contractul -d contractul
```

## Configuration

All configuration comes from the environment. `.env.example` lists every variable the
stack reads, with defaults safe for local development.

- **`.env` is never committed.** It is gitignored. Treat its contents as secrets even
  locally, because the habit is what protects you when the values are real.
- **Adding a variable means updating `.env.example`** in the same commit. Someone else's
  stack breaks otherwise, and they will have no way to know why.
- The backend reads the environment in exactly one place, `app/core/config.py`. Nothing
  else calls `os.environ`.

## Resetting

```bash
docker compose down -v          # deletes volumes: database AND generated documents
docker compose up --build
docker compose exec backend alembic upgrade head
docker compose exec backend python -m app.scripts.seed
```

`-v` is what makes this a true reset — without it the old database survives and you have
merely restarted. It is also irreversible: local data is gone. That is fine for local
development and catastrophic anywhere else, so build the habit of reading the flag before
typing it.

## Troubleshooting

**Port already in use.** Something else holds 5173, 8000, or 5432 — often a local
PostgreSQL installation. Stop it, or change the host-side port in
`docker-compose.override.yml`.

**Backend cannot reach the database.** Inside Compose, services address each other by
service name. The host is `database`, not `localhost` — `localhost` inside a container is
that container. Check `DATABASE_URL`.

**Changes not appearing.** If it is a dependency change, rebuild. If it is a source change,
confirm the volume mount is present in `docker-compose.override.yml`. Frontend assets can
also be cached — try a hard refresh before assuming the stack is at fault.

**PDF conversion fails or hangs.** LibreOffice runs headless inside the backend container
and is the most fragile part of the stack. Check `docker compose logs backend`. If the
container was built before LibreOffice was added to the Dockerfile, rebuild without cache:
`docker compose build --no-cache backend`.

**Everything is broken and you cannot say why.** Reset (above). If a clean stack works,
the problem was local state; if it does not, the problem is in the committed
configuration, which is a real bug worth reporting rather than working around.

**Slow on Windows or macOS.** File-system sync across the container boundary is the usual
cause. On Windows, keep the repository inside the WSL 2 filesystem rather than under
`/mnt/c` — the difference is large.
